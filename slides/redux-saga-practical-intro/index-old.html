Redux Saga

redux middleware to manage side effects (e.g. async network call) in a declarative and synchronous style

at the core, it uses ES6 feature called Generators

The "Good" Part
de-coupled m-v-c, literally

The "Ugly" Part
learning curve, but once you get over it, you'll like it
sometime, it feels like "more work" to implement things, but this is in expense of long term maintainability
some doubts that you may encounter
  • should I put all logic into saga? including UI logic? share about implementation of filter component/state. ans: who should manage or concern about this piece of data? is it only the component? or some other components also need it?
  • where should I put this piece of logic? component lifecycle? saga? redux store? action? share about the experience on setting up intercom where it depends on an authentication state (updates are managed by the saga, so the saga is the source of truth on this) and DOM manipulation (only accessible within react lifecycle, which is not available within saga). At the end we implemented within the react lifecylce with more conditional checks at componentWillRecieveProp to ensure that it only happens when authentication state is changed.

Verdict?
if it's a simple app, without much requirement, or simple CRUD apps, stick with what you are comfortable with
if you are maintaining bigger apps with lots of pieces that need to work together, consider using redux saga
